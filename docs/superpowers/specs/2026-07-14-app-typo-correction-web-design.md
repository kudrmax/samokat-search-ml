# Дизайн: веб-приложение «Исправление опечаток» (шаг 1)

Дата: 2026-07-14
Ветка: presentation

## Цель и контекст

Демо для презентации мастерской. Пользователь вводит поисковый запрос → приложение
показывает исправленный запрос с подсветкой изменённых слов. Главное — наглядность
на защите.

Это первый шаг большего пайплайна `query → исправление опечаток → два классификатора`.
Классификаторы (sklearn) подключаются **позже** и в этот дизайн намеренно не закладываются
(YAGNI). Но бэкенд-логика отделена от фронтенда, чтобы шаг с классификаторами добавлялся
без переделки.

## Корректор — используем присланный код КАК ЕСТЬ

Источник: `samokat/typos/ispravlator.ipynb`. Функцию-исправлятор **не улучшаем и не меняем**.

- `corrector.py` содержит функции `correct_word_prefix(...)` и `correct_query_prefix(...)`
  перенесённые из ноутбука **дословно** (включая загрузку словаря, `marisa_trie.Trie`,
  `SymSpell`, константу `DISTANCE_PENALTY = 10`).
- Поверх них — тонкий класс-адаптер `SymSpellCorrector`, который НЕ меняет логику, а
  переиспользует её:
  - конструктор принимает путь к словарю и однократно строит `word_freq`, `dictionary`,
    `trie`, `sym_spell` (тяжёлая инициализация — один раз при старте приложения);
  - метод `correct(query: str) -> list[WordCorrection]` разбивает запрос по словам
    (`query.strip().split()`) и для каждого слова вызывает существующую
    `correct_word_prefix(word)`, собирая пары `(original, corrected)`.
    Это ровно то, что делает `correct_query_prefix` внутри (`' '.join(correct_word_prefix(w) ...)`),
    только с сохранением пословного соответствия для подсветки.
- `corrector` не знает про HTTP/фронтенд. Путь к словарю: `samokat/data/domain_dictionary.txt`,
  задаётся относительно корня проекта и передаётся в конструктор.

Особенность, которую учитываем: корректор может **разбить слипшееся слово**
(`укропбатон` → `укроп батон`), поэтому `corrected` в одной паре может содержать пробел.
Подсветка строится по флагу `changed = (original != corrected)`.

## Модели (pydantic)

```python
class CorrectRequest(BaseModel):
    query: str = Field(min_length=1)

class WordCorrection(BaseModel):
    original: str
    corrected: str            # может содержать пробел (разбиение слипшихся слов)

    @property
    def changed(self) -> bool:
        return self.original != self.corrected

class CorrectResponse(BaseModel):
    original: str             # исходный запрос
    corrected: str            # склеенный исправленный запрос
    words: list[WordCorrection]
```

## Поток данных

1. Пользователь вводит запрос → `app.js` шлёт `POST /api/correct {"query": "кросчовки укропбатон"}`.
2. `main.py` → `SymSpellCorrector.correct(query)` →
   `[WordCorrection("кросчовки","кроссовки"), WordCorrection("укропбатон","укроп батон")]`.
3. Ответ:
   ```json
   {
     "original": "кросчовки укропбатон",
     "corrected": "кроссовки укроп батон",
     "words": [
       {"original": "кросчовки", "corrected": "кроссовки"},
       {"original": "укропбатон", "corrected": "укроп батон"}
     ]
   }
   ```
4. `app.js` рендерит две строки — оригинал и исправленный; слова с `changed=true` подсвечены.

## Интерфейс (простой, под перекраску позже)

Одна страница по центру:
- заголовок;
- поле ввода запроса + кнопка «Проверить» (Enter тоже отправляет);
- блок результата: строка «Запрос» (оригинал) и строка «Исправлено» с подсвеченными
  изменёнными словами;
- состояние загрузки и аккуратное сообщение об ошибке.

Вёрстка нейтральная и минималистичная — весь визуал в `style.css`, чтобы потом заменить
на фирстиль без правок логики.

## Ошибки

- Пустой/пробельный запрос → 422 (валидация pydantic `min_length=1`).
- Исключение внутри корректора → 500 с телом `{"error": "..."}`; фронт показывает сообщение,
  форму не ломает.

## Тесты

Согласно ML-стилю проекта тесты держим по минимуму. Но контракт слоёв проверяем:
2–3 юнит-теста на `SymSpellCorrector.correct` с известными примерами из ноутбука
(`крсовки` → `кроссовки`, `укропбатон` → `укроп батон`, неизменяемое слово остаётся как есть).
Функции-исправлятора при этом не меняются.
