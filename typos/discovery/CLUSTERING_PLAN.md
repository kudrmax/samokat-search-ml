# Дизайн: кластеризация типов искажений в запросах

**Дата:** 2026-07-07
**Ветка:** `typos-research`
**Deliverable:** `typos/discovery/typos_clustering.ipynb` — запускаемый сверху вниз ноутбук
с пояснениями по ходу. Результат — таксономия типов дефектов + таблица «запрос → тип».

## Цель

Мы имеем «чистый» датасет и **не знаем, какие типы искажений** есть в запросах.
Задача — **discovery**: обнаружить естественные группы дефектов (опечатка / слипшиеся
слова / оборванный запрос / латиница / раскладка / мусор / другой язык / не про еду)
и разложить запросы по ним. Это напрямую кормит **шаг 1** файла `typos/PLAN.md`
(разметка запросов ярлыками → «кандидаты на исправление»).

## Ключевое проектное решение

Типы искажений различаются **структурой текста**, а не смыслом. Семантические
эмбеддеры (e5, sbert) склеивают запросы по ТЕМЕ («напитки» vs «косметика»), а не по
типу дефекта, поэтому они здесь не подходят. Работаем на **символьно-лексических
признаках**.

Козырь: **каталог** (`item_name` + названия категорий) даёт словарь «правильных»
товарных слов. По нему меряем, настоящее слово или искажённое, и находим ближайший
эталон для подсказки исправления.

## Выбранный подход (вариант B)

Кластеризация по **именованным структурным признакам** (не по эмбеддингам).
Даёт discovery + полную интерпретируемость: видно, ПОЧЕМУ запросы попали в один
кластер. Отвергнутые альтернативы:

- **A. Только правила** — требует заранее знать типы, противоречит discovery.
- **C. Char-n-gram эмбеддинги → UMAP → HDBSCAN** — кластеры труднее объяснить,
  частично утекает тема. Оставляем как **опциональную визуальную перепроверку**
  (scatter «на посмотреть»).

## Пайплайн

### Шаг 0 — Словарь-эталон из каталога
Токенизировать `item_name` + все `categoryN_name` (lowercase, split по не-буквам) →
множество товарных слов с частотами. `pymorphy3` — как проверка «это слово русского
языка» для слов вне каталога.

### Шаг 1 — Структурные признаки на каждый уникальный запрос
Символьно-лексические сигналы (интерпретируемые, именованные):
- длина в символах, число токенов, средняя/максимальная длина токена;
- доля латиницы, кириллицы, цифр; наличие пунктуации;
- доля «известных» токенов (в каталоге ИЛИ в pymorphy);
- для неизвестных токенов — минимальное нормированное расстояние Левенштейна до
  ближайшего слова каталога (близко → опечатка, далеко → мусор/иностранное);
- флаги: смешанный алфавит в одном токене (раскладка), повтор буквы (`йооу`),
  оборванный хвост (короткий неизвестный последний токен, `багет чем`), only-latin.

### Шаг 2 — Отсев «ок»
Запрос считается чистым, если все значимые токены известны каталогу/pymorphy →
откладывается в бакет `ок` (с подсчётом количества). Кластеризуется только
**подозрительный остаток**. Бакет `ок` попадает в финальную таблицу с ярлыком `ок`.

### Шаг 3 — Кластеризация остатка
`sklearn.cluster.HDBSCAN` по стандартизованному вектору признаков.
Обоснование выбора (в ноутбуке): число типов неизвестно → не KMeans; плотностный
алгоритм сам выделяет бакет «шум» (выбросы).

### Шаг 4 — Интерпретация и имена кластеров
На каждый кластер: топ-репрезентативные запросы + средний профиль признаков +
авто-подсказка имени по доминирующим признакам. Человек (Макс) читает и утверждает
ярлыки. Строится маппинг `cluster_id → человекочитаемый тип`.

### Шаг 5 — Выход
- `typos/discovery/query_types.csv`: `query → тип` (`ок` + обнаруженные типы) +
  ключевые признаки + подсказка исправления (ближайшее слово каталога), где применимо;
- bar-chart распределения типов;
- UMAP/t-SNE scatter, раскрашенный по кластерам (визуальная перепроверка, вариант C).

## Зависимости

Из коробки: `pandas`, `numpy`, `scikit-learn` 1.9 (`sklearn.cluster.HDBSCAN` встроен),
`pymorphy3`, `matplotlib`, `seaborn`.
Доустановить в venv: **`rapidfuzz`** (быстрый Левенштейн для поиска ближайшего слова
каталога; лёгкие чистые wheels). **UMAP** опционален — при неудаче установки
заменяется на `sklearn` t-SNE/PCA.

## Границы (YAGNI)

- Не строим авто-исправлятор (это шаг 2 `PLAN.md`, отдельная задача).
- Не трогаем классификатор релевантности (шаг 3 `PLAN.md`).
- Не используем семантические эмбеддеры.
- Кластеризуем **уникальные** запросы (9 496), а не все 79 774 строки.
